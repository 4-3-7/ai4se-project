> **作者**：丁灵

---

## 一、Superpowers 技能：哪些真正有用，哪些流于形式

本次项目强制使用了 Superpowers 的七个技能。在我实际体验中，**writing-plans 发挥了最大作用**。brainstorming 产出的 SPEC 虽然完整，但真正让项目落地的是 writing-plans——它把模糊的"做一个 harness"分解为 27 个具体 task，每个 task 明确了目标、涉及文件、实现要点和验收标准。这种颗粒度的计划文档不仅为 subagent 提供了清晰的执行指引，也让我作为项目管理者能在任何时候快速定位进度和问题。没有这份 PLAN，靠我自己在脑海中追踪 27 个任务的依赖关系和完成状态，几乎不可能。

**test-driven-development 同样不可或缺**。AI 在编写测试用例时覆盖边界条件的能力远超我自己——它会自然地想到空输入、null 值、超时、并发等场景，而我手动写测试往往会遗漏。先让 AI 生成失败测试，再让它写代码通过测试，这个流程显著提升了交付代码的正确率和我的信心。重构环节也因为 AI 而变得轻松——不再需要小心翼翼地修补旧测试，直接让 AI 重新生成即可。

至于其他技能，brainstorming 的追问环节确实帮我澄清了几个关键决策（语言、主攻维度、分发方案），但它的价值更多体现在"结构化的决策记录"而非"启发新思路"。requesting-code-review 和 finishing-a-development-branch 在单人项目中发挥有限——没有真正的同行评审，review 更像是自检清单。这些技能的设计初衷面向团队协作，在单人项目中难免"形式大于实质"。

## 二、TDD 在 AI 协作下：阻碍还是放大器

我的结论是：**TDD 在 AI 协作下是放大器，而非阻碍。**

传统 TDD 的痛点在于写测试本身耗时——开发者需要手动构造测试数据、覆盖边界条件、维护测试代码。而 AI 消除了这个瓶颈。在我的项目中，subagent 能在几秒内为一个函数生成 10+ 个测试用例，覆盖正常路径、异常路径和边界条件。这意味着"先红"这一步的成本急剧下降，而"后绿"的收益不变——代码正确性得到保障。

更有意思的是重构环节。传统 TDD 中，重构意味着同时修改实现和测试，两者需要保持同步，这常常让人望而却步。但在 AI 辅助下，我直接让 AI 按照新接口重写测试，再把实现代码对齐过去。测试不再是重构的负担，反而成了重构的安全网。

不过这个结论有一个前提：**AI 写的测试必须是正确的**。我在项目中确实遇到过 AI 写了"永远通过"的测试（断言写错了），这需要人工 review。TDD 的红→绿流程本身也帮助发现了这类问题——如果测试一开始就是绿的，说明测试本身有问题。

## 三、Subagent 工作流：自主运行与偏离

整个项目通过 7 个 feature branch / worktree 并行推进，每个 worktree 对应一个独立模块。subagent 在获得了明确的 task 描述后，基本能自主完成实现和测试，不需要我中途干预。

但这里有一个关键前提：**PLAN 的质量决定了 subagent 的执行质量。** 冷启动验证暴露了这一点——当一个陌生的 agent 面对颗粒度过大的 T1.1（Mock + 两个真实 Provider 塞在一个 task 里），它选择了"完成能做的那一半"，跳过了需要 API Key 的另一半。这不是 agent 的错，而是我的 task 拆分不够细。好的 task 应该有"单一可独立验证的产出物"，粒度控制在 2-5 分钟。拆分到位的 task，subagent 几乎不会跑偏；拆分不到位的，agent 就会自行判断边界，结果不可控。

对于 task 颗粒度，我的经验是：**一个 task = 一个源文件 + 一个测试文件**，这是最优粒度。超过这个范围，agent 的行为就会变得不可预测。

---

## 四、SPEC 质量如何影响实现质量

冷启动验证是本次项目中最有价值的环节。我让一个与主开发 agent 完全不同的 agent，仅凭 SPEC + PLAN 去实现 1-2 个 task。结果暴露了三个具体问题：

1. **T1.1 颗粒度过大**：一个 task 包含 Mock LLM + 两个真实 Provider，agent 完成了 Mock 部分（17 个测试），但跳过了真实 Provider。原因是 SPEC 没有区分"纯代码任务"和"需要 API Key 的集成任务"。
2. **`provider.ts` 职责模糊**：SPEC 说它是"抽象接口"，但实际接口定义在 `types.ts`，agent 把它当成了 barrel 文件。
3. **隐性假设没有被文档化**：主 agent 和我在 brainstorming 中形成了大量共享上下文（如"Mock 优先，真实 Provider 延后"），但这些没有写进 SPEC，导致第二个 agent 只能看到字面意思。

这让我深刻认识到：**SPEC 的"完整性"不等于"清晰性"。** 一份 11 章的 SPEC 可能在文件级别描述上仍然不够精确。验证 SPEC 质量的唯一标准，就是让一个完全不了解背景的人（或 agent）仅凭文档去实现，看他在哪里受阻。

## 五、最有效的 Prompt 策略

我采用了"身份—任务—格式—约束"四段式 prompt 结构，并结合 PLAN 中的 task 描述：

- **身份**："你是一个擅长 TypeScript 后端开发的高级工程师"
- **任务**：PLAN 中的 task 描述（目标、涉及文件、实现要点）
- **格式**：指定输出形式（代码 + 测试 + 验证步骤）
- **约束**：保持编码风格统一、修改最小化、不改动指定模块

这种策略之所以有效，在于"身份"为 LLM 锚定了角色的技术栈和编码习惯，"任务"提供了明确的目标和验收标准，而"约束"则划定了行为边界——特别是"不改动 xx 模块"这一条，在多人/多 agent 并行开发中避免了意外的交叉修改。

## 六、凭证与分发：工程要求倒逼的思考

凭证安全要求（不硬编码、不进 Git、不写日志）促使我在审计日志模块中专门实现了 API Key 脱敏逻辑——正则匹配 `sk-ant-*`、`sk-proj-*`、`Bearer` 模式并替换为 `[REDACTED]`。这层防御如果不在需求中明确要求，大概率会被遗漏。

分发方面，Docker 多阶段构建迫使我思考用户的生产环境到底是什么样的——最终选择了 `node:22-alpine`、非 root 用户运行、volume 挂载工作目录。这些细节在纯本地开发中根本不会考虑，但恰恰是"别人能不能把你的项目跑起来"的关键。

最初 SPEC 中设计的凭证存储方案是系统 Keychain（keytar），但实际实现时改用了 AES-256-GCM 加密文件。这个变更源于对跨平台兼容性的重新审视——Keychain 在无桌面 Linux 上不可用，在 Windows 上也存在集成问题。加密文件方案虽然安全性略低，但胜在零依赖、跨平台一致。这是一个"工程现实打败理想设计"的典型案例。

## 七、如果重做，我会改变什么

如果重新开始，我会在 brainstorming 阶段更慎重地考虑两个方向：

一是**主攻维度**。治理（Governance）确实是工程深度最大的维度，但它的"可演示性"偏向防御——拦截危险命令、展示审计日志。如果选择反馈闭环作为主攻，可以做更主动的演示：agent 自动运行测试、自动分类失败、自动修复、自动验证，形成一个完整的"自我修正"循环，展示效果会更直观。

二是**项目范围**。当前项目是一个纯 CLI 工具 + Web 终端的形态。如果重做，可能会考虑直接做一个 Web 应用，将 agent 的治理面板、HITL 审批界面、审计日志浏览器、反馈报告都做成可视化组件，而非全部依赖终端输出。这会让项目在"应用可访问的 WebUI"这个要求上更有说服力。

## 八、对 Superpowers 方法论的批判

Superpowers 的核心假设是：**通过 spec-driven、subagent-built、TDD-guarded 的流程，可以让 AI 辅助的软件开发变得可控和可预测。** 在我的项目中，这个假设部分成立，部分不成立。

**成立的部分**：TDD 确实是 AI 协作的最佳实践——它给 AI 提供了明确的"对/错"信号，避免了 AI 在模糊需求中随机猜测。写作 Plan 再执行的方式也有效降低了 AI 的偏离概率。

**不成立或需要修正的部分**：

1. **"spec 写清楚了，subagent 就能执行到位"——这个假设过于乐观。** 冷启动验证证明，spec 和现实之间总有 gap。无论写得多详细，总会有隐性假设没有被文档化。这意味着 spec 不是一次性的产物，而是在执行过程中持续修订的活文档。

2. **"开发者愿意花大量时间在前置设计上"——这个假设在时间压力下不成立。** 我在 brainstorming + spec + plan 上花费了相当比例的时间，这对一个 27-task 的项目来说是合理的。但如果项目规模更大、时间更紧，这种前置投入是否还能维持？Superpowers 没有给出"最小可行 spec"的指导。

3. **"code review 是两阶段流程"——在单人项目中缺乏真正的 adversarial review。** 没有同行评审，review 本质上是自检，而人对 AI 生成的代码天然存在"看起来都对"的确认偏差。Superpowers 需要为单人项目提供更有效的 review 机制（如引入第二个 agent 作为 reviewer）。

4. **Superpowers 假设所有 task 都适合 subagent 模式**。但实际中，集成任务（如把治理系统集成到主循环）需要理解多个模块的上下文，subagent 在缺乏全局视野的情况下容易出错。这类任务更适合由人主导、AI 辅助。

---

## 总结

这个项目让我对"**当 LLM 能完成大部分编码工作时，工程师的价值在哪里**"这个问题形成了自己的答案：**工程师的价值不在写代码，而在定义"什么是对的"**——写出清晰的 spec、拆分合理的 task、判断 AI 产出的代码是否正确、设计安全边界和反馈机制。Seahorse 这个 harness 项目本身就是一个隐喻：LLM 是 CPU，只负责产生下一步设想；而 harness 是把这种设想转化为可靠系统的工程封装。这正是我作为工程师的核心价值所在。
